01 / Executive summary

What I built and why it matters for the role.

Money Model Architect is a CLI-backed advisor for Alex Hormozi's $100M Money Models. A human asks for business advice, the agent decides which operation is appropriate, and local tools handle state, calculations, retrieval, and trace recording.

Agent workflow

The agent chooses among saved-state reads, local-doc inspection, calculations, source-material search, clarification, context updates, and final answer synthesis.

RAG evaluation

Retrieval is measured only after tool-use judgment and source-need generation are evaluated, which keeps backend comparisons tied to real product behavior.

Production discipline

The repo records traces, reports, cache behavior, latency, quality metrics, known misses, and the current decision for each experiment.

The goal of the project is senior engineering judgment: show how the system is shaped, how quality is measured, where the evidence is strong, and where more production work remains.

02 / Architecture

The system design has four explicit layers.

The product is designed as an agent-operated workflow over a deterministic local tool layer. The important pieces are the runtime agent, the CLI boundary, durable business state, and the retrieval/eval infrastructure that makes answers auditable.

03 / Advisor workflow

A turn produces an answer and an audit trail.

The advisor flow is designed around the actions a real business conversation needs: context, math, source evidence, and memory. The saved trace lets a reviewer inspect the path the agent took.

1

Start session

session start loads saved business state, recent traces, known and missing facts, available operations, and trace requirements.

2

Choose operations

The agent can read the snapshot, inspect local docs, ask a clarifying question, update accepted facts, calculate economics, diagnose constraints, search source material, or read logs.

3

Retrieve source evidence

When the answer needs source support, the agent provides a structured source need with intent, corpus layers, focus terms, and optional constrained query variants.

4

Finish session

session finish records actions, source events, cited chunks, answer metadata, and the current snapshot.

Corpus layers

Unit economics, offers, upsells, downsells, and continuity. Cross-cutting chapters can belong to multiple layers.

Source event shape

Each search records the source need, query, retrieved chunks, citation IDs, and metadata needed for regression checks.

Business state

Facts provided by the user or found in local docs are saved so the agent can stop rediscovering the same context.

04 / Golden dataset suite

The evals target specific product risks.

The suite is organized as a golden dataset: cases, labels, scorers, reports, current results, and decisions. The order matters because retrieval quality depends on the agent choosing the right operation and source objective first.

Evaluation Ladder

3. Query Quality

5. Support

Check whether cited chunks support the answer's required claims and recorded source events.

05 / Retrieval experiments

The RAG work moved from baseline to candidate architecture.

The retrieval story started with a local lexical baseline, tested chunking, added embedding-backed vector search, compared BM25/vector/hybrid behavior, then added constrained query variants and Pinecone parity behind the same vector-store boundary.

Retrieval Pipeline

from agent-selected source need to citation-ready chunks
SourceNeed

Agent records intent, corpus layers, focus terms, user turn, and optional constrained query variants.

Query Builder

Deterministic flattened query stays as fallback; variants add causal and framework-specific phrasings.

Retrievers

BM25, vector, and hybrid search over the same heading-aware chunk set.

Fusion

Reciprocal-rank fusion combines variant results without requiring raw lexical and vector scores to match.

Evidence

Returned chunks include IDs, chapter/layer metadata, scores, previews, and citation fields.

Query-variant cost

Generated variants use 4.0 queries per case and produced 120 vector searches across the 30-case slice.

Warm-cache behavior

Recent vector and hybrid variant runs reported 100.0% corpus and query embedding cache hit rates.

Latency caveat

Hosted-vector p50 retrieval latency was roughly 5.3s for hybrid and 5.5s for vector in the current sequential harness.

06 / Observability and failure analysis

The system records enough evidence to debug the workflow.

The repo records the artifacts needed to explain both successes and misses: session traces, source events, cited chunks, snapshot diffs, Markdown reports, case-level JSONL, summary JSON, cache hit rates, latency metrics, and estimated embedding cost.

Failure categories

Misses are classified by layer: tool choice, source need, query construction, retrieval backend, trace shape, or final answer quality. That keeps fixes targeted.

Regression process

Blind acting-agent failures become golden cases when they reveal repeated or high-risk behavior. Source-event hardening is the clearest example: it moved from a miss to a 6-case regression set.

07 / Current decisions and next work

The next work is production hardening.

The current architecture is strong enough for a portfolio demo and useful enough to discuss in an interview. The next work is focused on the gaps that matter for a production version.

Current decisions

  • Use heading-aware chunks as the default local chunking strategy.
  • Keep BM25 as the lexical baseline and regression control.
  • Treat hybrid retrieval plus constrained query variants as the candidate product path.
  • Keep local vector search as the fast eval baseline.
  • Keep Pinecone behind the vector-store boundary for hosted retrieval.

Next production work

  • Reduce or parallelize query-variant fanout to improve hosted-vector latency.
  • Expand the golden datasets beyond the current portfolio-scale slices.
  • Separate cold-cache embedding cost from warm-cache steady-state reporting.
  • Add more answer-quality evaluation around final grounded recommendations.
  • Build a thin web surface over the same advisor and retrieval core.

Interview discussion points: why the boundary is shaped this way, why retrieval comparisons were staged behind tool-use evals, how the Pinecone latency result changes the roadmap, and how the current golden dataset would expand before production.
