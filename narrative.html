Building a Money Model Advisor

An AI advisor built on Alex Hormozi's $100M Money Models: it diagnoses a founder's unit economics and answers with cited passages from the book. This page documents how it was designed and measured.

Objective

A founder asks for business advice in plain language. The advisor diagnoses their unit economics, teaches the relevant Money Models concepts, and recommends the next change to test — with every claim that needs support cited to the book, and every turn recorded as an auditable trace.

Design

An agent plans each turn; a deterministic CLI persists typed business state, runs the math, and executes retrieval. The candidate search path is hybrid (keyword + vector, merged into one ranking) over heading-aware chunks, with agent-written query variants, cached embeddings, and a vector store interface with two implementations: local for development and evals, Pinecone for hosted retrieval.

Method

Every architecture choice was decided by measurement: nine golden datasets, ordered so agent judgment is evaluated before the retriever, with the judgment suites scored against agents that never saw the expected answers. Anything more complex than the baseline had to beat the baseline on a measurement to be adopted; what didn't was rejected and recorded.

01
Objective

Plain-language business questions in; a diagnosis, a recommended next test, and cited source passages out.

What the advisor does

Take a founder's plain-language question, diagnose their money model, and recommend the next change to test — citing the book wherever a claim needs support.

The questions arrive as "I think CAC is too high," "cash is tight," "what should I add after the first sale?" Those are different advisory moves — diagnosis, teaching, offer design, a next-step recommendation — and the system has to decide which one a turn needs before any retrieval question matters.

1.1 · ScopeWhere this is deliberately overbuilt

Three parts of this system — the vector database, semantic retrieval, and namespace partitioning — are more infrastructure than an advisor over one book needs. They were built anyway, because the target role builds production RAG infrastructure and each one demonstrates part of that work.

Vector database
The design started from two hand-built documents: a decision tree distilled from the book's coaching logic and a summary of its frameworks (both in corpus/). For an advisor over this one book, those two documents would probably be sufficient — no vector database required. They still shaped the system: the decision tree inspired the five-layer namespace division, and the frameworks summary seeded the golden eval set. The Pinecone-backed pipeline exists to demonstrate production RAG infrastructure (§2.4, §3.6).
Semantic retrieval
BM25 alone would normally be enough here — but only because of how the queries are built. The system's generated queries carry the book's exact framework terms ("CFA," "rollover upsell"), and when query and corpus share vocabulary, keyword search is hard to beat. Queries phrased the way real users talk would erode that advantage, which is why the eval suite audits query realism separately (§3.1). Vector and hybrid retrieval were built to demonstrate semantic-retrieval engineering, and were promoted only after beating the BM25 control (§3.4–3.5).
Namespace partitioning
One namespace with metadata filters is the right default at 202 chunks. The five-layer Pinecone split was built and benchmarked anyway, because the role calls for operating multiple Pinecone namespaces. The result: identical retrieval, slower worst-case latency — so the single namespace stays the default (§3.7).
02
Design

The agent/CLI split, a versioned instruction file, typed business state, eight CLI tools, a RAG pipeline with generated queries, and a validated trace for every turn.

The design

2.1 · Agentic workflowAn AI agent driving a local CLI

The first architectural decision, and the one everything else follows from: the agent makes every judgment call, a local CLI does everything deterministic and repeatable, and nothing calls a hosted model at runtime.

Agent — semantic judgment

  • decide the next advisory move
  • inspect local docs when context is missing
  • decide whether source material is needed at all
  • generate SourceNeed objects + query variants
  • judge whether retrieved chunks support a claim
  • synthesize the final answer
  • decide which facts are safe to save

CLI — deterministic support

  • create, locate, and persist advisor state
  • run unit-economics formulas reproducibly
  • execute retrieval, local or Pinecone
  • expose chunks with stable IDs and scores
  • validate and record traces + source events
  • generate eval reports

One advisory turn, with the split in action:

human asks the agent for advice → agent runs session start loads saved state, missing fields, history → agent inspects local business docs only if saved context is missing → agent saves accepted facts to the BusinessSnapshot (§2.2) → agent runs CLI tools calculation · diagnosis · source search → agent answers citing retrieved chunks where claims need support → agent runs session finish validates and persists the turn trace (§2.5)

This division of labor is what the rest of the page depends on: every judgment the agent makes lands in a recorded trace where it can be audited and scored (§2.5), and everything the CLI does is testable as ordinary code.

Why a local CLI, run by a subscription agent

The deterministic side is a local CLI, and the agent driving it is a subscription coding tool (Codex), not a hosted model API. The runtime makes no hosted model calls — a constraint chosen for cost: the agent runs on a subscription, so the only per-token charge anywhere in the build-and-eval loop is embeddings, cached after first use (§2.4). The CLI shape pays off three ways: the agent gets commands it can run reliably, the eval harness gets output stable enough to score exactly, and a developer can rerun any turn by hand.

The skill

The agent's instructions are one versioned skill file (SKILL.md, 238 lines): how to run a turn, when to search versus ask, the query variants each search requires (§2.4), and what the trace must contain at the end. It's treated like code — versioned in git and amended through the test-fix loop in §3.9: when one agent misreads an instruction the skill is patched, and when two fail the same way the case becomes a regression test.

Reading the founder's business files in place

Running locally also lets the agent read the founder's own business directory — their real files on disk. Pointed at that directory (--business-dir), the agent reads those files when saved context is missing and records each accepted fact back to its source path and content hash (§2.2). A hosted service would have to copy the directory in first; the CLI just reads it where it lives.

Building a hosted web product before the advisor loop and the eval method were proven would have been premature. If one is built later, it is a web front end over the same core (§4) — the stable JSON contract and the vector-store boundary are what make that cheap. The only outside services today are the embedding API, for vectorization, and the Pinecone vector store (§2.4).

2.2 · Data integrationBusinessSnapshot: typed business state, every field tracked to its source

The advisor's memory of a business is one typed object — identity, offer stack, economics, problem framing — and every field records where it came from.

V1 stores only what's needed to decide the next advisory action, run diagnostics, target retrieval, and explain what's missing before a recommendation. The whole schema:

Block by block:

business
Who the business is: business_type, icp (ideal customer profile), delivery_model. These change how the book's examples apply and how fulfillment cost and margin should be read.
money_model
The book's five offer types — attraction offer, core offer, upsell, downsell, continuity — each stored as {exists, description, price}. exists alone wouldn't be enough — "add an upsell" is bad advice for a business that has one but priced it too low — so the description and price are stored with it. And exists takes three values: true, false only when there's evidence the offer is really absent, and null when nobody knows yet.
economics
The unit-economics numbers: cac (customer acquisition cost), first-30-day and recurring gross profit, gross margin, lifetime gross profit, and the derived payback_period_months — recomputed whenever inputs change.
problem
The user's goal, their symptoms in their own words (reported_symptoms: "cash is tight"), and the advisor's conclusions (diagnosed_constraints: slow_payback) — separate fields, so what the user reported is never confused with what the advisor concluded.
advisor_state
Status fields the CLI computes and the agent can read but not invent: which diagnoses the saved numbers can support, the still-missing facts — which is what drives the clarifying questions — and which sections of the book are likely relevant. The agent decides whether any of this matters for the current turn. The headline field, advisory_status, moves through four stages:
insufficient_context→ diagnosable→ diagnosed→ recommendable
field_sources
One record per saved fact, naming where it came from (file, conversation, or calculated) with a confidence grade and timestamp. Facts taken from a file keep the file's path and a hash of its contents, so the fact is marked stale when the file changes. The record for a CAC read from a metrics file: 
"economics.cac": {
  "source_type": "file",
  "source": "metrics/q1-unit-economics.csv",
  "source_hash": "abc123",
  "confidence": "high",
  "updated_at": "2026-06-05T00:00:00Z"
}

Merge rules are explicit — inspected evidence beats inference, direct user statements beat weak file inferences, newer direct user statements beat older ones — and conflicts are preserved in the trace rather than silently overwritten.

The snapshot is also the cache for business context: facts are accepted once and reused every turn after — the advisor remembers the business, advice traces to sourced facts, and docs aren't re-inspected every turn. Full schema, field rationale, readiness and merge rules: BUSINESS_SNAPSHOT_V1.md.

2.3 · Tool useEight CLI operations, mapped to one advisory turn

The CLI (§2.1) emits structured JSON for the agent; a project skill hides the flags and payloads from humans. Its eight operations map onto one advisory turn.

session start
Prepare the turn: load the snapshot; show advisory status, known and missing facts, retrieval hints, recent turns, available operations, and what must be recorded at the end. It does not answer, decide the next action, or search.
snapshot / snapshot set
Read and persist accepted facts. Validates field paths, recomputes readiness; never infers facts or decides which matter.
calculate
Deterministic formulas — CAC, gross profit, gross margin, lifetime gross profit, payback period, CFA (client-financed acquisition) level. The agent must not do this math from memory.
diagnose
Deterministic unit-economics diagnosis from saved numbers. The agent decides whether the result is relevant and how to explain it.
search
Retrieve citation-ready chunks from an agent-written SourceNeed, against a selectable retrieval strategy and vector store (§2.4).
session finish
Validate and persist the completed turn as one structured artifact: message pair, actions, source events, calculation events, cited chunk IDs, ending snapshot.
logs
Inspect prior turns when the conversation depends on earlier context.
index pinecone
Index management: upsert corpus vectors into the hosted vector index (§2.4).

2.4 · RAG pipelineHybrid retrieval over the chunked book

Every search starts from a structured request the agent writes — a SourceNeed — and runs hybrid: two rankers over the same chunked book, keyword (BM25) and meaning-based (vector), combined into one ranked list.

The corpus: 32 lesson transcripts in five namespaces

The source material is the book's lessons — 32 transcript files, one per lesson, about 88,000 words. The lessons are partitioned into five namespaces, and each search names which to look in:

unit-economics offers upsells downsells continuity

The request: a SourceNeed

A SourceNeed says what a search is for. It carries an intent, the namespaces to look in, focus terms, the user's turn for context, and the query variants described next. Different advisory jobs need different evidence, so each search names its intent:

teaching_evidence diagnostic_evidence comparison_evidence recommendation_evidence

A turn can be mixed — an answer may both teach and recommend — but each search does one job. An answer that needs two kinds of evidence runs two searches and records two source events.

Query generation: agent-written variants + a deterministic fallback

Queries are generated, not lifted from the user's message. The agent writes 2–4 variants of each search — the same need phrased different ways, typically once in cause-and-effect terms and once in the book's framework vocabulary. One variant is always the deterministic fallback: the focus terms joined mechanically, with no agent judgment in it. All variants run, and their results are merged by rank, so a badly phrased variant can't sink the search. A real request from a recorded turn:

Why not one freeform agent query? Traceability: every variant is recorded in the trace, and the fallback gives every search a stable, judgment-free baseline to score against. The variants exist because of a measured failure: when searches ran only the mechanical query, a question about why fulfillment cost matters retrieved the wrong passages — a flat list of terms had lost the "why" (§3.5).

How a search runs

Chunking
Heading-aware: preserve transcript sections, fall back to fixed windows. 202 chunks with stable IDs and citation metadata.
Search
BM25 and embedding-based vector search run over the same chunks, and their results are merged by position, not score — reciprocal-rank fusion, the reranking step. Merging by raw score would mean calibrating two unrelated scales against each other, a common source of fragile hybrid search; merging by rank sidesteps it.
Embeddings
OpenAI text-embedding-3-small embeds chunks and queries. One model so far — the 3-small vs 3-large comparison is queued (§4).
Results
Each search returns the top 5 chunks (CLI default --top-k 5), each with a stable ID, its score, and citation metadata. The agent reads them and decides which actually support the claim — rank is never treated as truth.
Cost control
Corpus and query embeddings are cached under .cache/embeddings/; warm eval runs make zero external embedding API batches.
Storage
A narrow VectorStore interface with two implementations: local in-memory for fast dev and evals, and Pinecone for hosted retrieval. The first Pinecone adapter is a small REST client rather than the official SDK — the interface is what matters; the adapter behind it can move to the SDK without touching advisor or eval logic. The index also supports a five-namespace layer layout; a single namespace with metadata filters is the default (§3.7).

None of these was a default. Section 3 is the methodology behind each one: how it was measured, and why it was kept or rejected.

2.5 · ObservabilityEvery turn ends as one validated trace

Each turn ends as one validated artifact. session finish enforces its shape rather than trusting the agent's self-report:

Actions
A non-empty list of known operation labels. Zero source events is valid — not every turn should search.
Source events
Each must carry the full SourceNeed, the executed queries, and inspected chunks with IDs and scores.
Citations
Every cited chunk ID must appear in a source event, unless the trace explicitly marks it as an external, non-corpus citation — the agent cannot quietly cite material it never retrieved. Chunks inspected but not cited only warn: that's legitimate behavior. This is the grounding rule: a claim about the book either cites an inspected chunk or isn't made as a book claim at all — when retrieval finds no support, the advisor has to say so instead of inventing a citation.
Calculations
If actions include calculate, the trace must include structured calculation_events with metric, inputs, and output — added after agents were observed claiming math their traces couldn't prove (§3.9).
State
The ending snapshot is captured automatically.

One trace answers, mechanically: what the agent did, what it searched for and why, what it inspected, cited, and computed, and what state it left behind. Every eval in §3 reads these traces.

03
Method

Nine golden datasets scored against blind acting agents. Agent judgment gated the retrieval comparisons; each design choice was adopted, rejected, or deferred from a measurement.

Method

3.1 · Evaluation frameworkNine golden datasets, five layers

The evals are layered so a failure points at one component. Agent judgment is tested before query writing, and query writing before the retriever — so when two retrieval strategies differ, the difference is about retrieval, not a bad query or a wrong decision upstream.

  1. Tool-use judgment Should this turn search source material at all — versus read state, inspect docs, calculate, clarify, or answer directly?
  2. Source-need generation When search is right, what retrieval objective, corpus layers, and focus concepts should guide it?
  3. Search-query quality Does query construction retrieve citeable chunks for that source need?
  4. Strategy comparison Only now do BM25-vs-vector-vs-hybrid comparisons measure retrieval architecture.
  5. Support coverage Do the cited chunks actually support the claims the answer must make?

Five of the nine datasets sit directly on these layers — tool use, source need, search queries, the strategy comparison built on them, and the support-coverage labels. The other four check the eval system itself: two verify traces record what actually happened (source events, calculation events), one compares chunking strategies, and one audits whether eval queries are phrased the way real users talk. Every dataset pairs with an automated scorer that writes an auditable report. The acting agents in the judgment evals run on Codex. OpenAI is used only to turn text into vectors for search — no hosted model makes any judgment that gets scored.

Blind, three roles
An acting agent performs each case using the skill and CLI without seeing expected labels. A trace extractor maps its observable behavior — commands run, files read, snapshot diffs — into recorded actions, keeping weak evidence visible as inferred or missing. A separate scorer compares those actions to the labels. Self-report never becomes the metric.
Label strength
"Do not search on this turn" is strict: searching fails the case. "This chunk is known useful" is a starting label, not a complete list of every useful chunk. Exact term matching is only a debugging signal — "front-end offer" fails a text match against "front end offer" — so the measure that counts is whether the concepts are covered, judged with a written rationale. Misses against the starting labels get reviewed before they're counted.

3.2 · Quality scoringDoes the agent act on the right turns?

The tool-use suite has 24 cases. Most were captured during the build as development and regression traces; the rest are held-out scenarios, run after the instructions were frozen, by acting agents that never saw the expected answers. The source-need suite has 14 cases: 10 where the right move is to search, 4 controls where it isn't. Results: 23 of 24 tool-use cases passed on strict first-action accuracy — 24 of 24 after one documented label correction — with zero false and zero missed searches. Source-need scored 100% on the search/no-search decision and matched the expected namespaces exactly on 9 of the 10 search cases. The misses, case by case:

Corrected label
One held-out case required reading logs as the literal first action. Review concluded reading the snapshot first was harmless context-loading — logs were still read before answering — and the label records that review explicitly.
Unresolved miss
On the free-trial case the agent picks the offers layer without also adding downsells. It remains open.
General rules, not patches
When a case disagreed with its label, the resolution became a general rule, not a one-case patch. Two examples: payment plans are labeled downsells — they look like pricing, but their function is reducing purchase friction. And a turn asking how to fix payback gets recommendation_evidence, not diagnostic_evidence: the user already wants a fix, so the search exists to support the recommendation, even though "payback" sounds diagnostic.
Decision

Agent judgment is solid enough to build the retrieval comparisons on.

With search/no-search decisions at 100% on the captured sets, differences between retrieval strategies now measure retrieval — not tool-use noise.

3.3 · Chunking strategyHeading-aware kept; framework-aware rejected

Two chunking strategies were compared on a labeled set. Heading-aware splits the transcripts at their section headings. The framework-aware variant adds split points at cue phrases like "how it works," "step one," and "examples" — the spots where a lesson switches from naming a framework to explaining it — on the theory that tighter chunks would match definition-style and mechanics-style queries better. It didn't help: identical Hit@1 (81.25%), and an MRR gain of +0.0041.

Decision

Keep heading-aware chunking.

A gain of less than half a percent doesn't justify maintaining a second chunking pipeline; the comparison stays in the repo as a recorded non-adoption.

3.4 · Hybrid retrieval + rerankingComparing BM25, vector, and hybrid

BM25 served as the keyword baseline and control, not the product architecture. Its job was to reveal whether vector and hybrid retrieval actually beat the simple thing, and it set a high bar — the generated queries carry the book's exact framework terms ("CFA," "rollover upsell," "payback period"), and that shared vocabulary makes keyword retrieval unusually competitive. Against it: vector search and rank-fused hybrid, all over the same chunks.

The golden set itself grew with the work. The first version of query generation was the deliberately simple baseline: one query built from the saved business state, before SourceNeeds existed. The eval exposed it — nearly every search produced the same broad diagnostic query, and only half the cases found a useful chunk (50.0%). That measurement is what created the structured SourceNeed (§2.4). The case set was then expanded from 10 to 30, and chunks the new searches surfaced were reviewed and added to the labels where they were genuinely citeable.

The first comparison favored the control: vector and hybrid each missed a case at Hit@5 that BM25 found. Several other apparent misses were label-set limitations — retrieved chunks on menu upsells, rollover upsells, waived fees, and continuity bonuses were directly citeable and were promoted into the known-useful labels after inspection. searchq_v1_001 survived review as a true miss. Dashes in these tables are metrics the source reports don't include.

Decision

No retriever change from this table alone.

Vector and hybrid each missed a case the keyword control found, and that miss traced to how the query was built — not to which retriever ran it. Fixing it is the next section.

3.5 · Query generationOne flattened query lost to agent-written variants

The searchq_v1_001 miss is what put query variants into the design (§2.4):

The case
The user asks why fulfillment cost matters for paid ads — a causal question.
The diagnosis
The v1 query builder collapsed the agent's source need into one fixed list of its terms — a reasonable baseline, since it kept agent judgment out of the comparison and gave every eval the same query to score. But a list of nouns loses the cause-and-effect shape of the question, so vector search ranked nearby payback/CAC chunks above the one passage that actually explains gross profit and fulfillment cost. The failure was in how the query was built, not in the retriever.
The fix
Deliberately constrained — not one freeform agent query, which would trade away traceability. The agent writes 2–4 query variants in a fixed format, the mechanical query stays in the set as a fallback, and all results are merged with reciprocal-rank fusion so no variant can push the fallback out.

Cost reported beside quality: variants average 4.0 queries per case — 120 vector searches across the set. On warm runs the embedding cache hit 100% for corpus and query vectors, so the vector and hybrid evals made zero external embedding API batches.

Decision

Hybrid retrieval + constrained variants is the candidate product path. BM25 stays as the control.

Variants fixed the miss, but they add another agent-authored artifact that can be wrong — which is why the deterministic fallback and the golden set stay in place as guards, and why this is a candidate default rather than a final claim.

3.6 · Latency benchmarksPinecone matched local quality, and exposed slow retrieval

Local retrieval came first so iteration and evals stayed fast. Once the strategy was justified, the same 30-case set was rerun against the Pinecone implementation of the VectorStore interface (§2.4): 100% Hit@3/Hit@5, mean useful rank 1.17, 100% query-cache hit rate, zero embedding API batches. Quality reproduced exactly.

What the latency benchmark exposed instead: the first hosted run executed each case's four query variants one at a time, and p50 retrieval landed around 5.3s for hybrid and 5.5s for vector — versus sub-millisecond local BM25. Running the variants in parallel (8 workers) brought the same 30-case benchmark to roughly 1.4s p50 and 1.8s p95, quality unchanged.

Decision

Hosted retrieval works: Pinecone returns the same results as the local store, and parallelizing the variant queries fixed the worst of the latency.

Top-k tuning and request-level observability are still open — no claim of production latency readiness before they're done.

3.7 · Pinecone namespacesOne namespace or five

The JD names multi-namespace Pinecone operation, and the corpus already had a five-layer taxonomy — so the layout was tested rather than assumed. The corpus was indexed into five layer namespaces, with the agent selecting which to query, and the same 30 cases, variants, cache, and hybrid path ran in both conditions: one namespace with metadata filters versus the five-way split routed from each case's expected layers — better routing than any production router could promise.

The two conditions retrieved identically. Not just the same scores: across all 90 case-and-retriever rows, the same top-5 chunks in the same order. Namespaces would only help if wrong-layer chunks were crowding out right-layer ones, and on this corpus they are not. Latency got worse where it counts: p50 barely moved (~1.45s vs ~1.43s), but p95 went from ~1.76s to ~3.01s, on 140 vector searches instead of 120.

The mechanism: at 202 chunks, a Pinecone query is dominated by the network round-trip and fixed per-request overhead, not vector scan time — so searching a smaller namespace saves microseconds. The split instead turns one round-trip into up to five for multi-layer cases, and a case finishes only when its slowest namespace answers. Waiting on the slowest of five queries is usually slower than waiting on one — which is exactly what the p95 shows.

One scope caveat: this set is saturated — Hit@5 is 100% for every backend in both conditions — so it cannot detect a quality improvement from anything. The supported claim is "no measurable difference on this set," not "namespaces never help." If the corpus grows to where layers interfere in ranking, re-test on a harder set.

Decision

Namespace support stays in; a single namespace with metadata filters stays the default.

The five-way split is implemented and verified, loses on tail latency, and wins nothing on quality here. It gets promoted only on a measured win.

3.8 · Model routing & tieringCan a cheaper model match the planning tier?

The JD asks for routing that improves unit economics while keeping output quality. The expensive tier is already named by the product: planning runs as Claude in the CLI, and the recorded traces of that path score 92.9% strict pass on source-need and 100% strict pass on tool-use under the same deterministic scorers used everywhere else in this report. The question this section answers is whether any cheaper tier matches that baseline on the same suites.

Four hosted OpenAI tiers were replayed as single bounded JSON completions with the needed state inlined in the prompt: gpt-5, gpt-5-mini, gpt-5-nano, and the non-reasoning gpt-4.1-mini. 152 completions, zero request errors, every prompt and raw response committed under evals/runs/model_routing/. Source-need is the cleaner discriminator — its task is one bounded completion, so the comparison isolates model judgment from harness — and is the table shown below.

No tested OpenAI tier matches the Claude-CLI baseline, and the failure direction depends on the family. gpt-5 keeps the search/no-search call perfect but drops 30 points of intent match. The cheaper reasoning tiers fail closed: gpt-5-mini declines 5 of 10 expected searches, gpt-5-nano declines 9 of 10. Answers from those tiers would silently lose their citations. The non-reasoning tier fails open: gpt-4.1-mini searches on 3 of the 4 no-search controls and returns malformed action objects on 3 of 24 tool-use cases. Cheap reasoning is also not free in latency — gpt-5-nano spends 45,599 completion tokens on tool-use, triple gpt-5-mini, and its p50 lands near gpt-5's.

Two caveats are explicit. (1) The Claude row was recorded interactively through the CLI with state inspection and multi-step iteration; the OpenAI rows were single API calls with state inlined. Some of the gap is harness, not model — source-need is the cleaner discriminator because its task fits one bounded completion, which is why it's the table above. Six tool-use cases failed on every API tier because their labels expect CLI actions (read_snapshot, calculate, search_source_material) that a single completion with state already inlined will skip. (2) Claude Haiku and Sonnet have not been replayed in this harness yet. That is the same-provider downgrade test and the obvious next experiment; it is named here as a gap, not folded into the v1 decision.

Decision

Nothing routes downward for planning in v1. Claude-in-CLI stays the planning tier; the real unit-economics lever stays the deterministic CLI, which costs zero model tokens per call.

The first routed downgrade candidates are bounded low-stakes transforms — query-variant phrasing first — and any candidate, OpenAI or Claude Haiku, must match the recorded baseline on the relevant golden suite before it ships.

3.9 · Regression detectionFailures became regression suites

Beyond the planned experiments, the design was hardened by a blind test-fix loop: send realistic turns to acting agents, inspect the saved traces — not just the answers — and fix the layer that actually failed (skill guidance, CLI command design, trace schema, or eval case). The promotion rule is explicit: one agent confused by instructions — patch the skill. Two agents failing the same way — regression case. Anything that could corrupt state or hide unsupported advice — CLI guard after a single occurrence.

Trace shape → schema guard
Agents listed calculate in actions while traces didn't preserve the math. Fix: structured calculation_events, required whenever calculate is claimed (§2.5). The 5-case regression suite passes 5/5 — and one agent that nested the events under metadata was correctly rejected by the validator, confirming the guard catches the real failure class.
Source events → regression suite
On a "what should we fix first?" turn, the correct answer needs two recorded searches: diagnostic evidence plus recommendation evidence for the chosen fix. Blind runs failed twice, two different ways. After tightening the guidance, the case became a 6-case set covering multi-search, pure-diagnosis, pure-recommendation, no-search, teaching-only, and continuity turns — with counter-cases in both directions so the rule against extra searches can't be passed by simply never searching. Current: 6/6, zero extra-search warnings.
Path plumbing → CLI guard
One acting agent's shell-variable mistake pointed session start at the advisor repo itself. Fix: the skill shows a safe shell pattern, and the CLI refuses to treat the advisor repo as a business directory unless explicitly overridden.
Command design → command change
A realistic offer-sequencing turn exposed that diagnose expected raw JSON instead of reading saved state, and that agents skipped writing query variants. Fix: diagnose --business-dir reads the snapshot directly, and query variants became mandatory — the skill requires 2–4 agent-written variants per search, session finish rejects source events without them, and a second 6/6 source-event pass verified the stricter contract.
04
Limits & next

Golden sets are portfolio-scale, hosted latency is unsolved, answer-quality coverage is thin, and per-turn token cost is unmeasured.

What this doesn't prove, and what comes next

Scale
Golden sets are portfolio-scale (5–65 cases each). They drive architecture decisions and catch the failure modes found so far; they are not production benchmarks. The high pass rates read as "the design clears everything that has broken it" — not "nothing will break it."
Hosted latency
~1.4s p50 / ~1.8s p95 for Pinecone hybrid after parallelizing the variant queries (down from ~5.3s sequential). Hosted quality matches local exactly; top-k tuning and request-level observability remain (§3.6).
Answer quality
The thinnest coverage in the eval suite. Decisions, traces, and retrieval are well-tested; grounded final-recommendation quality is the next dataset to grow.
Cost reporting
Embedding cost, cache hit rates, and per-run query counts are instrumented. Agent-side token usage isn't — the agent runs inside a subscription coding tool, which doesn't expose per-turn token counts. Where calls do go through an API directly, exact token and cost numbers are reported: the embedding runs (§3.5) and the model-routing experiment (§3.8).
Model routing
Generation tiering is now measured (§3.8): no tested downgrade held planning quality, so the routing decision is recorded rather than open. Still missing: embedding model selection, and a controlled same-harness cross-provider run — the recorded Claude reference is interactive, the OpenAI tiers answered single API calls.

The product path forward would reuse what's built: a thin web surface over the same advisor core. That is why retrieval sits behind a vector-store interface, the CLI emits stable JSON, and the advisor logic never learned whether its vectors were local or hosted.
