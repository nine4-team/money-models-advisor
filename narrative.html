Building a Money Model Advisor

An AI advisor built on Alex Hormozi's $100M Money Models: it diagnoses a founder's unit economics and answers with cited passages from the book. This page documents how it was designed and measured.

Objective

A founder asks for business advice in plain language. The advisor diagnoses their unit economics, teaches the relevant Money Models concepts, and recommends the next change to test — with every claim that needs support cited to the book, and every turn recorded as an auditable trace.

Design

An agent plans each turn; a deterministic CLI persists typed business state, runs the math, and executes retrieval. The candidate search path is hybrid (keyword + vector, rank-fused) over heading-aware chunks, with agent-written query variants, cached embeddings, and a Pinecone-backed vector store behind a swappable interface.

Method

Every architecture choice was decided by measurement: nine golden datasets, ordered so agent judgment is evaluated before the retriever, with the judgment suites scored against blind acting-agent runs. Choices that didn't earn their complexity were rejected and recorded.

01
Objective

Plain-language business questions in; a diagnosis, a recommended next test, and cited source passages out.

What the advisor does

Take a founder's plain-language question, diagnose their money model, and recommend the next change to test — citing the book wherever a claim needs support.

The questions arrive as "I think CAC is too high," "cash is tight," "what should I add after the first sale?" Those are different advisory moves — diagnosis, teaching, offer design, a next-step recommendation — and the system has to decide which one a turn needs before any retrieval question matters.

1.1 · ScopeWhere this is deliberately overbuilt

Three parts of this system — the vector database, semantic retrieval, and namespace partitioning — are more infrastructure than an advisor over one book needs. They were built anyway, because the target role builds production RAG infrastructure and each one demonstrates part of that work.

Vector database
The design started from two hand-built documents: a decision tree distilled from the book's coaching logic and a summary of its frameworks (both in corpus/). For an advisor over this one book, those two documents would probably be sufficient — no vector database required. They still shaped the system: the decision tree inspired the five-layer namespace division, and the frameworks summary seeded the golden eval set. The Pinecone-backed pipeline exists to demonstrate production RAG infrastructure (§2.4, §3.5).
Semantic retrieval
BM25 alone would normally be enough here — but only because of how the queries are built. The system's generated queries carry the book's exact framework terms ("CFA," "rollover upsell"), and when query and corpus share vocabulary, keyword search is hard to beat. Queries phrased the way real users talk would erode that advantage, which is why the eval suite audits query realism separately (§3.1). Vector and hybrid retrieval were built to demonstrate semantic-retrieval engineering, and were promoted only after beating the BM25 control (§3.4).
Namespace partitioning
One namespace with metadata filters is the right default at 202 chunks. The five-layer Pinecone split was built and benchmarked anyway, because the JD names multi-namespace operation and the split exercises index management. The test came back honest: identical retrieval, worse tail latency — so the single namespace stays the default (§3.6).
02
Design

Five decisions: an agent/CLI split, typed business state, eight CLI tools, hybrid retrieval, and a validated trace for every turn.

The design

2.1 · Agentic workflowAn AI agent driving a local CLI

The first architectural decision, and the one everything else follows from: the agent makes every judgment call, a local CLI does everything deterministic and repeatable, and nothing calls a hosted model at runtime.

Agent — semantic judgment

  • decide the next advisory move
  • inspect local docs when context is missing
  • decide whether source material is needed at all
  • generate SourceNeed objects + query variants
  • judge whether retrieved chunks support a claim
  • synthesize the final answer
  • decide which facts are safe to save

CLI — deterministic support

  • create, locate, and persist advisor state
  • run unit-economics formulas reproducibly
  • execute retrieval, local or Pinecone
  • expose chunks with stable IDs and scores
  • validate and record traces + source events
  • generate eval reports

One advisory turn, with the split in action:

human asks the agent for advice → agent runs session start loads saved state, missing fields, history → agent inspects local business docs only if saved context is missing → agent saves accepted facts to the BusinessSnapshot (§2.2) → agent runs CLI tools calculation · diagnosis · source search → agent answers citing retrieved chunks where claims need support → agent runs session finish validates and persists the turn trace (§2.5)

This division of labor is what the rest of the page depends on: every judgment the agent makes lands in a recorded trace where it can be audited and scored (§2.5), and everything the CLI does is testable as ordinary code.

Why a local CLI, run by a subscription agent

The deterministic side is a local CLI, and the agent driving it is a subscription coding tool (Codex), not a hosted model API. That shape pays off three ways. Running locally was required anyway, since no hosted model is allowed in the runtime. A CLI gives the agent commands it can run reliably, gives the eval harness output stable enough to score exactly, and lets a developer rerun any turn by hand. And because the agent runs on a Codex subscription rather than a metered generation API, the only per-token charge anywhere in the build-and-eval loop is embeddings — cached after first use (§2.4).

Reading the founder's business files in place

Running locally also lets the agent read the founder's own business directory — their real files on disk. Pointed at that directory (--business-dir), the agent reads those files when saved context is missing and records each accepted fact back to its source path and content hash (§2.2). A hosted service would have to copy the directory in first; the CLI just reads it where it lives.

Building a hosted web product before the advisor loop and the eval method were proven would have been premature. If one is built later, it is a second surface over the same core (§4) — the stable JSON contract and the vector-store boundary are what make that cheap. The only outside services today are the embedding API, for vectorization, and the Pinecone vector store (§2.4).

2.2 · Data integrationBusinessSnapshot: typed business state, every field tracked to its source

The advisor's memory of a business is one typed object — identity, offer stack, economics, problem framing — and every field records where it came from.

V1 stores only what's needed to decide the next advisory action, run diagnostics, target retrieval, and explain what's missing before a recommendation:

Business identity
Type, ICP, delivery model — they change how examples apply and how fulfillment cost and margin should be read.
Money-model stack
Five positions — attraction offer, core offer, upsell, downsell, continuity — each stored as {exists, description, price}. The flag alone is not enough: "add an upsell" is bad advice when one exists but is underpriced. null means unknown; false requires evidence of absence.
Economics
CAC (customer acquisition cost), first-30-day gross profit, recurring gross profit, gross margin, lifetime gross profit, and the derived payback period — recomputed whenever inputs change.
Problem framing
The user's goal, their symptoms in their own words ("cash is tight"), and the advisor's stored conclusions (slow_payback) — kept separate so the advisor doesn't optimize the wrong thing.

Every non-null field records its source: source type (file / conversation / calculated), the path and content hash for file-derived facts so they invalidate when the file changes, a confidence grade, and a timestamp. Merge rules are explicit — inspected evidence beats inference, direct user statements beat weak file inferences, newer direct user statements beat older ones — and conflicts are preserved in the trace rather than silently overwritten.

The snapshot also computes deterministic hints. The advisory status moves through a four-state machine:

insufficient_context→ diagnosable→ diagnosed→ recommendable

Readiness flags (payback diagnosis needs CAC and first-30-day gross profit; stack diagnosis needs to know which positions exist), missing-field lists that drive targeted clarifying questions, and candidate retrieval layers all live here too. The agent decides whether any hint matters for the current turn.

The snapshot is also the cache for business context: facts are accepted once and reused every turn after — the advisor remembers the business, advice traces to sourced facts, and docs aren't re-inspected every turn.

2.3 · Tool useEight CLI operations, mapped to one advisory turn

The CLI (§2.1) emits structured JSON for the agent; a project skill hides the flags and payloads from humans. Its eight operations map onto one advisory turn.

session start
Prepare the turn: load the snapshot; show advisory status, known and missing facts, retrieval hints, recent turns, available operations, and what must be recorded at the end. It does not answer, decide the next action, or search.
snapshot / snapshot set
Read and persist accepted facts. Validates field paths, recomputes readiness; never infers facts or decides which matter.
calculate
Deterministic formulas — CAC, gross profit, gross margin, lifetime gross profit, payback period, CFA (client-financed acquisition) level. The agent must not do this math from memory.
diagnose
Deterministic unit-economics diagnosis from saved numbers. The agent decides whether the result is relevant and how to explain it.
search
Retrieve citation-ready chunks from an agent-written SourceNeed, against a selectable retrieval strategy and vector store (§2.4).
session finish
Validate and persist the completed turn as one structured artifact: message pair, actions, source events, calculation events, cited chunk IDs, ending snapshot.
logs
Inspect prior turns when the conversation depends on earlier context.
index pinecone
Index management: upsert corpus vectors into the hosted vector index (§2.4).

2.4 · RAG pipelineHybrid retrieval over the chunked book

Every search starts from a structured request the agent writes — a SourceNeed — and runs hybrid: two rankers over the same chunked book, keyword (BM25) and meaning-based (vector), combined into one ranked list.

The request: a SourceNeed

A SourceNeed says what a search is for. It carries an intent, the corpus layers to look in, focus terms, the user's turn for context, and 2–4 query variants. Different advisory jobs need different evidence, so each search names its intent:

teaching_evidence diagnostic_evidence comparison_evidence recommendation_evidence

A turn can be mixed — an answer may both teach and recommend — but each search does one job. An answer that needs two kinds of evidence runs two searches and records two source events.

The corpus: five retrieval namespaces

The corpus is the book's lessons, one file per lesson, partitioned into five namespaces — the corpus layers. A search is scoped to one or more of them, and the agent names them in its SourceNeed:

unit-economics offers upsells downsells continuity

How a search runs

Chunking
Heading-aware: preserve transcript sections, fall back to fixed windows. 202 chunks with stable IDs and citation metadata.
Search
BM25 and embedding-based vector search run over the same chunks; their rankings are combined with reciprocal-rank fusion — the reranking step — so raw keyword and vector scores never need to be directly comparable.
Queries
The variants are causal and framework phrasings of the same source need, fused with a deterministic flattened fallback query so they can't crowd out the stable baseline.
Cost control
Corpus and query embeddings are cached under .cache/embeddings/; warm eval runs make zero external embedding API batches.
Storage
A narrow VectorStore interface with two implementations: local in-memory for fast dev and evals, and Pinecone for hosted retrieval. The first Pinecone adapter is a small REST client rather than the official SDK — the interface is what matters; the adapter behind it can move to the SDK without touching advisor or eval logic. The index also supports a five-namespace layer layout; a single namespace with metadata filters is the default (§3.6).

None of these was a default. Section 3 is the methodology behind each one: how it was measured, and why it was kept or rejected.

2.5 · ObservabilityEvery turn ends as one validated trace

Each turn ends as one validated artifact. session finish enforces its shape rather than trusting the agent's self-report:

Actions
A non-empty list of known operation labels. Zero source events is valid — not every turn should search.
Source events
Each must carry the full SourceNeed, the executed queries, and inspected chunks with IDs and scores.
Citations
Every cited chunk ID must appear in a source event, unless the trace explicitly marks it as an external, non-corpus citation — the agent cannot quietly cite material it never retrieved. Chunks inspected but not cited only warn: that's legitimate behavior.
Calculations
If actions include calculate, the trace must include structured calculation_events with metric, inputs, and output — added after agents were observed claiming math their traces couldn't prove (§3.8).
State
The ending snapshot is captured automatically.

One trace answers, mechanically: what the agent did, what it searched for and why, what it inspected, cited, and computed, and what state it left behind. Every eval in §3 reads these traces.

03
Method

Nine golden datasets scored against blind acting agents. Agent judgment gated the retrieval comparisons; each design choice was adopted, rejected, or deferred from a measurement.

Method

3.1 · Evaluation frameworkNine golden datasets, five layers

Each layer isolates a failure mode the layers below it can't explain — so retrieval comparisons measure retrieval, not upstream noise.

  1. Tool-use judgment Should this turn search source material at all — versus read state, inspect docs, calculate, clarify, or answer directly?
  2. Source-need generation When search is right, what retrieval objective, corpus layers, and focus concepts should guide it?
  3. Search-query quality Does query construction retrieve citeable chunks for that source need?
  4. Strategy comparison Only now do BM25-vs-vector-vs-hybrid comparisons measure retrieval architecture.
  5. Support coverage Do the cited chunks actually support the claims the answer must make?

Five of the nine datasets sit directly on these layers — tool use, source need, search queries, the strategy comparison built on them, and the support-coverage labels. The other four check the eval system itself: trace integrity (source events, calculation traces), the chunk inventory (chunking comparison), and query realism. Every dataset pairs with an automated scorer that writes an auditable report. Acting agents in the judgment evals are Codex-driven; vector search uses OpenAI embeddings, for vectorization only.

Blind, three roles
An acting agent performs each case using the skill and CLI without seeing expected labels. A trace extractor maps its observable behavior — commands run, files read, snapshot diffs — into recorded actions, keeping weak evidence visible as inferred or missing. A separate scorer compares those actions to the labels. Self-report never becomes the metric.
Label strength
"Do not search on this turn" is a strict behavioral expectation. "This chunk is known useful" is a non-exhaustive seed label. Exact focus-term recall is only a debugging signal — "front-end offer" fails a substring match against "front end offer" — so concept coverage judged with rationale is the semantic measure. Misses against seed labels get reviewed before they're interpreted.

3.2 · Quality scoringDoes the agent act on the right turns?

Twenty-four tool-use cases — dev and regression traces captured in-thread, held-out scenarios run after the prompts were frozen by separate acting agents that never saw the labels — and fourteen blind source-need cases (10 search, 4 no-search controls). Results: 23/24 strict first-action accuracy, 24/24 after one documented label correction, with zero false and zero missed searches; 100% search-decision accuracy, with layer exact match on 9 of the 10 search cases. The misses, case by case:

Corrected label
One held-out case required reading logs as the literal first action. Review concluded reading the snapshot first was harmless context-loading — logs were still read before answering — and the label records that review explicitly.
Unresolved miss
On the free-trial case the agent picks the offers layer without also adding downsells. It remains open.
Taxonomy, not patches
Layer disagreements produced general rules: payment plans belong to downsells because their function is reducing purchase friction; a fix-oriented turn keeps recommendation intent even when payback appears as a focus term.
Decision

Judgment is good enough to gate the retrieval comparisons.

With search/no-search decisions at 100% on the captured sets, differences between retrieval strategies now measure retrieval — not tool-use noise.

3.3 · Chunking strategyHeading-aware kept; framework-aware rejected

Two chunking strategies were compared on a labeled set: heading-aware scored 81.25% Hit@1 and 0.8917 MRR; a framework-aware variant scored identical Hit@1 and +0.0041 MRR.

Decision

Keep heading-aware chunking.

The gain doesn't clear the adoption threshold; the comparison stays in the repo as a recorded non-adoption.

3.4 · Hybrid retrieval + rerankingComparing BM25, vector, and hybrid

BM25 served as the keyword baseline and control, not the product architecture. Its job was to reveal whether fancier retrieval earns its complexity, and it set a high bar — the generated queries carry the book's exact framework terms ("CFA," "rollover upsell," "payback period"), and that shared vocabulary makes keyword retrieval unusually competitive. Against it: vector search and rank-fused hybrid, all over the same chunks.

The golden slice itself grew with the work. An earlier snapshot-only query mode scored 50.0% by repeating one broad diagnostic query — the failure that motivated structured source needs — and the slice was then expanded from 10 to 30 cases, with newly retrieved citeable chunks reviewed and added to the labels.

The first comparison favored the control: vector and hybrid each missed a case at Hit@5 that BM25 found. Several other apparent misses were label-set limitations — retrieved chunks on menu upsells, rollover upsells, waived fees, and continuity bonuses were directly citeable and were promoted into the known-useful labels after inspection. searchq_v1_001 survived review as a true miss. Dashes in these tables are metrics the source reports don't include.

That miss is what put query variants into the design (§2.4):

The case
The user asks why fulfillment cost matters for paid ads — a causal question.
The diagnosis
The v1 query builder flattened the agent's source need into a deterministic bag of terms — a reasonable baseline, since it isolated retrieval execution from agent judgment and gave every eval a stable query to score. But a flat noun list loses the causal shape of the question, so dense retrieval ranked adjacent payback/CAC chunks above the one passage that actually explains gross profit and fulfillment cost. The failure was in query generation, not the retriever.
The fix
Deliberately constrained — not one freeform agent query, which would trade away traceability. The agent writes query variants under a constrained schema, the deterministic flattened query stays as a fallback variant, and results are fused with reciprocal-rank fusion so early variants can't crowd out the fallback.

Cost reported beside quality: variants average 4.0 queries per case — 120 vector searches across the slice. On warm runs the embedding cache hit 100% for corpus and query vectors, so the vector and hybrid evals made zero external embedding API batches.

Decision

Hybrid retrieval + constrained variants is the candidate product path. BM25 stays as the control.

Variants fixed the miss, but they add another agent-authored artifact that can be wrong — which is why the deterministic fallback and the golden slice stay in place as guards, and why this is a candidate default rather than a final claim.

3.5 · Latency benchmarksPinecone matched local quality, and exposed slow retrieval

Local retrieval came first so iteration and evals stayed fast. Once the strategy was justified, the same 30-case slice was rerun against the Pinecone implementation of the VectorStore interface (§2.4): 100% Hit@3/Hit@5, mean useful rank 1.17, 100% query-cache hit rate, zero embedding API batches. Quality reproduced exactly.

What the latency benchmark exposed instead: the first hosted run executed each case's four query variants one at a time, and p50 retrieval landed around 5.3s for hybrid and 5.5s for vector — versus sub-millisecond local BM25. Running the variants in parallel (8 workers) brought the same 30-case benchmark to roughly 1.4s p50 and 1.8s p95, quality unchanged.

Decision

Pinecone proves the hosted path and preserves quality; parallelizing the variant queries fixed the worst of the latency.

Top-k tuning and request-level observability are still open — no claim of production latency readiness before they're done.

3.6 · Pinecone namespacesOne namespace or five

The JD names multi-namespace Pinecone operation, and the corpus already had a five-layer taxonomy — so the layout was tested rather than assumed. The corpus was indexed into five layer namespaces, with the agent selecting which to query, and the same 30 cases, variants, cache, and hybrid path ran in both conditions: one namespace with metadata filters versus the five-way split routed from each case's expected layers — better routing than any production router could promise.

The two conditions retrieved identically. Not just the same scores: across all 90 case-and-retriever rows, the same top-5 chunks in the same order. Namespaces would only help if wrong-layer chunks were crowding out right-layer ones, and on this corpus they are not. Latency got worse where it counts: p50 barely moved (~1.45s vs ~1.43s), but p95 went from ~1.76s to ~3.01s, on 140 vector searches instead of 120.

The mechanism: at 202 chunks, a Pinecone query is dominated by the network round-trip and fixed per-request overhead, not vector scan time — so searching a smaller namespace saves microseconds. The split instead turns one round-trip into up to five for multi-layer cases, and a case finishes only when its slowest namespace answers. The maximum of five draws from the same latency distribution has a worse tail than a single draw, which is exactly what the p95 shows.

One scope caveat: this slice is saturated — Hit@5 is 100% for every backend in both conditions — so it cannot detect a quality improvement from anything. The supported claim is "no measurable difference on this slice," not "namespaces never help." If the corpus grows to where layers interfere in ranking, re-test on a harder slice.

Decision

Namespace support stays in; a single namespace with metadata filters stays the default.

The five-way split is implemented and verified, loses on tail latency, and wins nothing on quality here. It gets promoted only on a measured win.

3.7 · Model routing & tieringCan a cheaper model do the planning?

The JD asks for model routing that improves unit economics while maintaining output quality. The second half is the constraint, so it was tested first: the source-need and tool-use golden suites were replayed as single bounded JSON completions against four hosted tiers — gpt-5, gpt-5-mini, gpt-5-nano, and the non-reasoning gpt-4.1-mini — and scored by the same deterministic scorers used on the recorded agent traces. 152 completions, zero request errors, every prompt and raw response committed as a run artifact.

No tested tier held quality. gpt-5 kept the search/no-search boundary perfect (100%) but still reached only 71.4% strict pass on source-need against 92.9% for the recorded interactive agent. The cheaper reasoning tiers failed closed: gpt-5-mini declined 5 of 10 expected searches, gpt-5-nano declined 9 of 10 — answers would silently lose their citations. The non-reasoning tier failed open: gpt-4.1-mini searched on 3 of the 4 no-search controls and returned malformed action objects on 3 of 24 tool-use cases. Model family changed the failure direction, not just the rate.

The economics were measured, and they are tempting: gpt-4.1-mini runs roughly 10× faster (p50 ~1s versus 5–14s) and ~20× cheaper per case than gpt-5. The quality columns are what disqualify it. Cheap reasoning is also not free: gpt-5-nano spent 45,599 completion tokens on the tool-use suite — triple gpt-5-mini — for equal-or-worse results, with latency near gpt-5's.

Caveat: six tool-use cases failed on every API tier, because their labels expect CLI actions that a bounded completion with the state already inlined will reasonably skip. Part of that suite's gap measures the harness, not the model; source-need is the cleaner tier discriminator. And the recorded-agent reference ran a different harness and provider (Claude operating the CLI interactively), so it is a direction of evidence, not a controlled cross-provider benchmark.

Decision

Nothing routes downward in v1. Planning stays on the strong interactive tier; the real unit-economics lever stays the deterministic CLI, at zero model cost.

First downgrade candidates are bounded low-stakes transforms like query-variant phrasing — gated on matching the recorded baseline on the relevant suite.

3.8 · Regression detectionFailures became regression suites

Beyond the planned experiments, the design was hardened by a blind test-fix loop: send realistic turns to acting agents, inspect the saved traces — not just the answers — and fix the layer that actually failed (skill guidance, CLI command design, trace schema, or eval case). The promotion rule is explicit: one agent confused by instructions — patch the skill. Two agents failing the same way — regression case. Anything that could corrupt state or hide unsupported advice — CLI guard after a single occurrence.

Trace shape → schema guard
Agents listed calculate in actions while traces didn't preserve the math. Fix: structured calculation_events, required whenever calculate is claimed (§2.5). The 5-case regression suite passes 5/5 — and one agent that nested the events under metadata was correctly rejected by the validator, confirming the guard catches the real failure class.
Source events → regression suite
On a "what should we fix first?" turn, the correct answer needs two recorded searches: diagnostic evidence plus recommendation evidence for the chosen fix. Blind runs failed twice, two different ways. After tightening the guidance, the case became a 6-case set covering multi-search, pure-diagnosis, pure-recommendation, no-search, teaching-only, and continuity turns — with counter-cases in both directions so the anti-over-search rule can't overfit. Current: 6/6, zero extra-search warnings.
Path plumbing → CLI guard
One acting agent's shell-variable mistake pointed session start at the advisor repo itself. Fix: the skill shows a safe shell pattern, and the CLI refuses to treat the advisor repo as a business directory unless explicitly overridden.
Command design → command change
A realistic offer-sequencing turn exposed that diagnose expected raw JSON instead of reading saved state, and that agents skipped writing query variants. Fix: diagnose --business-dir reads the snapshot directly, and query variants became mandatory — the skill requires 2–4 agent-written variants per search, session finish rejects source events without them, and a second 6/6 source-event pass verified the stricter contract.
04
Limits & next

Golden sets are portfolio-scale, hosted latency is unsolved, answer-quality coverage is thin, and per-turn token cost is unmeasured.

What this doesn't prove, and what comes next

Scale
Golden sets are portfolio-scale (5–65 cases each). They drive architecture decisions and catch the failure modes found so far; they are not production benchmarks. The high pass rates read as "the design clears everything that has broken it" — not "nothing will break it."
Hosted latency
~1.4s p50 / ~1.8s p95 for Pinecone hybrid after parallelizing the variant queries (down from ~5.3s sequential). Hosted quality matches local exactly; top-k tuning and request-level observability remain (§3.5).
Answer quality
The thinnest coverage in the eval suite. Decisions, traces, and retrieval are well-tested; grounded final-recommendation quality is the next dataset to grow.
Cost reporting
Embedding cost, cache hit rates, and per-run query counts are instrumented. Agent-side token usage isn't — the acting agent runs outside the API path in this prototype.
Model routing
Generation tiering is now measured (§3.7): no tested downgrade held planning quality, so the routing decision is recorded rather than open. Still missing: embedding model selection, and a controlled same-harness cross-provider run — the recorded Claude reference is interactive, the OpenAI tiers are bounded completions.

The product path forward would reuse what's built: a thin web surface over the same advisor core. That is why retrieval sits behind a vector-store interface, the CLI emits stable JSON, and the advisor logic never learned whether its vectors were local or hosted.

Integrate LLMs with internal systems, APIs, and data sources Typed business state with per-field source tracking and merge rules (§2.2), a bounded CLI tool surface (§2.3), structured retrieval requests (§2.4), and in-place reads of the firm's own files (§2.1).
RAG pipelines across Pinecone namespaces — chunking strategy, embedding model selection, hybrid retrieval, reranking A Pinecone-backed store behind a vector-store interface, with the corpus organized as a five-layer namespace taxonomy and the one-vs-five namespace layout benchmarked before choosing a default (§2.4, §3.6); chunking strategy decided by comparison (§3.3); hybrid retrieval with reciprocal-rank fusion as the recorded v1 reranking baseline, with the trigger for a true reranker written down: known-useful chunks landing in the top-20 candidates but missing the top-5 (§3.4). Embedding model selection is a named gap — one embedding model so far (§4).
Evaluation framework — golden datasets, automated quality scoring, retrieval metrics, latency benchmarks, regression detection Nine golden datasets, each with an automated scorer and a written report (§3.1); Hit@3 / Hit@5 / mean-rank retrieval metrics (§3.4); a hosted-vs-local latency benchmark (§3.5); two regression suites and a CLI guard promoted from real failures (§3.8).
Embedding strategies, index management, and retrieval tuning Deterministic, cached embeddings (§2.4); index pinecone upsert tooling over the vector-store interface (§2.3–2.4); the strategy comparison, variant fusion, query parallelization, and namespace-layout benchmark are measured retrieval tuning, with top-k tuning named as the next step (§3.4–3.6).
Reduce LLM API costs — model routing, caching, token management, or architectural improvements Subscription-operated agent over a local CLI, so the build-and-eval loop's only metered call is embeddings (§2.1); those are cached at 100% warm hit rate, zero API batches on reruns (§2.4, §3.4); BusinessSnapshot prevents repeated context discovery (§2.2).
Observability — cost-per-request, token usage, quality signals, anomaly detection Per-turn validated traces (§2.5); per-run cost, cache, and latency accounting (§3.4–3.5); scorer reports as quality signals (§3.1). Token usage and anomaly detection are named gaps (§4).
Model routing and tiering to improve unit economics; multiple LLM providers Measured (§3.7): four OpenAI tiers replayed two golden suites under the same scorers. No downgrade held planning quality, so nothing routes downward and the cost lever stays the deterministic CLI. Family tradeoffs are recorded — reasoning tiers fail closed, the non-reasoning tier fails open and breaks the output contract. A controlled same-harness Anthropic condition is the named next step.
Comfortable in both TypeScript and Python Python throughout this project; TypeScript is not exercised here.

Every number and design rule above has a source artifact in the repo: DESIGN.md is the canonical narrative, BUSINESS_SNAPSHOT_V1.md and CLI_DESIGN.md define the state and tool contracts, GOLDEN_DATASET.md maps the eval suite, and evals/reports/ holds the per-experiment reports, case files, and summaries.
